# Instructor Runbook

This runbook covers preparation, student enrollment, credential delivery, class operation,
revocation, and teardown.

## Access and trust model

There are three separate credentials:

| Credential | Generated by | Stored by | How it is delivered |
| --- | --- | --- | --- |
| Student WireGuard private key | Student | Student only | Never transmitted |
| Student WireGuard public key | Student | Instructor/Terraform | Private intake form or LMS |
| Mythic account password | Instructor/Mythic | Student and Mythic | Individually, preferably separate from the VPN profile |

Terraform generates a per-student `.conf` template containing a private-key placeholder. The
template is not usable until the student inserts their private key locally.

Never request student private keys. Never place private keys, completed VPN profiles, Mythic
passwords, server keys, or Terraform state in Google Sheets, Forms, email threads, chat rooms,
or source control.

## Recommended Google Workspace workflow

A shared editable Google Sheet is convenient but is the wrong intake mechanism: students can
alter rows, identity-to-key mappings are exposed to the class, and mistakes are difficult to
attribute. Use a Google Form that writes to an instructor-only response sheet instead.

Configure the form to:

- require sign-in with the course account, if available;
- collect the student's email address;
- limit each account to one response, while allowing edits before the deadline;
- restrict the response sheet to instructors and teaching assistants;
- avoid file uploads because only a single public-key text value is needed.

Suggested fields:

| Field | Required | Validation |
| --- | --- | --- |
| Full name | Yes | Match the roster |
| Course email | Yes | Collected automatically where possible |
| Student/lab ID | Yes | Lowercase identifier assigned by instructor |
| WireGuard public key | Yes | One line, 44 base64 characters, normally ending in `=` |
| Operating system | Yes | Windows, macOS, or Linux |

For Google Forms response validation, use the regular expression
`^[A-Za-z0-9+/]{43}=$`. Validation catches formatting errors but does not prove that the student
controls the corresponding private key; the first successful VPN handshake provides that check.

The response sheet may contain public keys and assigned VPN addresses because these are not
secret. Keep it instructor-only because the roster mapping is still course data.

If a visible class sheet is required, use it only for non-sensitive status such as `received`,
`profile sent`, and `connection verified`. Do not publish keys, IP addresses, usernames, or
passwords there.

## Before enrollment

1. Confirm written authorization, class dates, named instructors, and the permitted activity.
2. Confirm that the GCP project is dedicated to this lab.
3. Decide whether the exercise is only a Mythic interface exercise or requires a target.
4. If a target is required, provision it separately and document its exact address, permitted
   techniques, data-handling rules, and reset process. The current Terraform does not create one.
5. Publish [the student guide](STUDENT-ACCESS.md), the key-submission deadline, and support
   contact.
6. Create the private enrollment form and reconcile responses against the official roster.

## Collect and validate public keys

### Short version

Send this exact request to each student:

```text
Install WireGuard and generate a WireGuard key pair on your own computer.

Linux/macOS/WSL:
umask 077
wg genkey | tee student.key | wg pubkey > student.pub

Send me:
1. Your full name
2. Your course email
3. Your operating system
4. Only the single-line contents of student.pub

Keep student.key private. Do not send it, upload it, or paste it into a form.
```

When a response arrives:

1. open `terraform/terraform.tfvars`;
2. find the student's reserved entry;
3. paste the public key between the quotes in `public_key = ""`;
4. do not change the assigned name or VPN IP;
5. run the Terraform plan and apply commands below;
6. send the generated `.conf` template and Mythic login to that student individually.

The active roster contains:

```hcl
{
  name       = "student01"
  public_key = "" # Paste only the public key here.
  vpn_ip     = "10.20.100.11"
},
{
  name       = "student02"
  public_key = "" # Paste only the public key here.
  vpn_ip     = "10.20.100.12"
},
```

An empty `public_key` reserves the identity but does not create a WireGuard peer or client
profile. After a valid key is inserted and Terraform is applied, the profile appears as
`terraform/generated-clients/student01.conf` or `student02.conf`.

For each response:

1. verify the name and email against the roster;
2. remove accidental leading/trailing whitespace from the public key;
3. verify that the key has WireGuard's expected base64 form;
4. reject duplicate public keys;
5. assign a unique lowercase name and VPN address.

The current active deployment uses `10.20.100.10` for the instructor. Use
`10.20.100.11` onward for students. Maintain an instructor-only
roster like:

| Name | Email | Terraform name | VPN IP | Public key received | Profile sent | Mythic user | Verified |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Student Name | student@example.edu | `student01` | `10.20.100.11` | Yes | Yes | `student01` | No |

Add each peer to `terraform/terraform.tfvars`:

```hcl
students = [
  {
    name       = "student01"
    public_key = "WIREGUARD_PUBLIC_KEY_FROM_STUDENT"
    vpn_ip     = "10.20.100.11"
  }
]
```

Before applying, check duplicate names, addresses, and keys:

```bash
cd terraform
terraform fmt -recursive
terraform validate
terraform plan -out lab.tfplan
```

Review the plan carefully. A student-list change replaces the VPN VM and temporarily interrupts
all VPN connections, although its reserved public IP remains unchanged. Make roster changes
before class or during an announced maintenance window.

Apply the reviewed plan:

```bash
terraform apply lab.tfplan
terraform output
```

Client templates are written to `terraform/generated-clients/`.

## Create and deliver student access

For every enrolled student:

1. confirm their generated profile has the assigned name, VPN IP, current endpoint, server public
   key, `AllowedIPs = 10.20.0.0/24`, and the private-key placeholder;
2. create or enable one named Mythic operator account;
3. assign the operator only to the intended class operation;
4. generate a temporary, unique Mythic password;
5. record delivery and verification status without storing the password in the roster.

Send access individually through the institution's LMS or another authenticated per-student
channel. A practical two-message delivery is:

- LMS/private course message: the student's `.conf` template, internal URL, assigned Mythic
  username, authorized time window, and link to the student guide;
- separate channel or separate message: the temporary Mythic password.

Do not put all student profiles in one shared folder. Although generated templates contain no
student private key, individual delivery reduces confusion and prevents students from using the
wrong assigned VPN identity.

Suggested delivery message:

```text
Your assigned lab identity is student01.

1. Download student01.conf.
2. Replace REPLACE_WITH_STUDENT_PRIVATE_KEY with your own WireGuard private key.
3. Do not send or upload the completed file.
4. Connect and open the supplied internal URL.
5. Log in as student01 using the temporary password sent separately.
6. Use only the assigned operation, target, and class time.

Read STUDENT-ACCESS.md before connecting. Report problems without sending your private key or
completed VPN configuration.
```

## Prepare Mythic

After the initial deployment, retrieve the generated administrator credentials through IAP:

```bash
gcloud compute ssh rt-lab-team-server \
  --project PROJECT_ID \
  --zone ZONE \
  --tunnel-through-iap \
  --command='cd /opt/mythic && sudo grep "^MYTHIC_ADMIN_" .env'
```

Then:

1. connect through the instructor VPN profile;
2. open the internal redirector URL;
3. log in as the generated administrator and change its password;
4. create one named operator per student; never share the administrator account;
5. create a separate operation for the class or group;
6. limit each operator to the operation required for the exercise;
7. set the operation end date and document the cleanup decision.

Treat Mythic events, credentials, payloads, and student activity as sensitive course data.

## Pre-class checklist

- [ ] Authorization and scope are current.
- [ ] Student roster is reconciled with Terraform peers and Mythic users.
- [ ] All three instances are running.
- [ ] Mythic bootstrap has completed.
- [ ] The instructor VPN has a current handshake.
- [ ] The redirector returns the Mythic login page through the VPN.
- [ ] Direct student access to team-server port 7443 is blocked.
- [ ] Student-to-student VPN traffic is blocked.
- [ ] Public TCP/443 access is blocked.
- [ ] The assigned operation exists and operator membership is correct.
- [ ] Any exercise target is explicitly documented and reset to a known state.
- [ ] The stop time, incident contact, and evidence-submission process are published.

## Health checks

Use `terraform output admin_commands` for the exact IAP SSH commands.

Gateway:

```bash
sudo wg show
sudo systemctl status wg-quick@wg0
```

Redirector:

```bash
sudo nginx -t
sudo systemctl status nginx
```

Team server:

```bash
sudo tail -n 200 /var/log/redteam-lab-bootstrap.log
cd /opt/mythic
sudo ./mythic-cli status
```

From an enrolled VPN client:

```bash
curl -kI "$(terraform -chdir=terraform output -raw redirector_url)"
```

## During class

1. Keep an instructor connected to the lab and available for scope decisions.
2. Monitor WireGuard handshakes, Mythic activity, and relevant GCP logs.
3. Require students to identify their own activity and use named accounts.
4. Pause the exercise if there is unexpected cross-student visibility, an unknown target, or
   possible traffic outside scope.
5. Record account changes, target resets, incidents, and exceptions.
6. Do not troubleshoot by collecting a student's private key or completed VPN profile.

## Logging

VPC Flow Logs are enabled at 50% sampling. The student-to-redirector firewall rule is logged.
Cloud NAT records errors, GCP Audit Logs record control-plane changes, and Mythic and Nginx keep
application logs on their VMs.

Tell students what is logged and how long records are retained. Do not add log-scrubbing
automation; logs are needed for troubleshooting, accountability, and incident review.

## End-of-class checklist

- [ ] Tell students to log out and disconnect WireGuard.
- [ ] Disable Mythic accounts if access is not needed between sessions.
- [ ] Stop or reset any separately provisioned targets.
- [ ] Export only the evidence required by course policy.
- [ ] Review unexpected connections or events.
- [ ] Stop instances if the next class does not require persistence.
- [ ] Update the roster status and record any access that must be revoked.

Stopping instances reduces compute cost, but Cloud NAT and the reserved external IP can still
incur charges. Destroy the environment between cohorts when persistence is unnecessary.

## Remove a student

1. Disable or remove the student's Mythic operator account immediately.
2. Remove their object from `students` in `terraform/terraform.tfvars`.
3. Run `terraform plan` and `terraform apply` during a maintenance window.
4. Confirm their public key no longer appears in `sudo wg show` on the replacement gateway.
5. Remove their access artifacts from instructor delivery systems according to course policy.
6. Record the revocation time.

Removing a peer invalidates that student's private key for this VPN without requiring the
instructor to possess or delete the private key.

## Compromised credential response

Student WireGuard private key or completed profile:

1. disable the student's Mythic account;
2. remove the peer and apply Terraform;
3. have the student generate a new key pair;
4. enroll the new public key and issue a new template;
5. review logs for the affected VPN address and time window.

Mythic password:

1. disable the account or reset its password;
2. terminate active sessions if supported;
3. review that operator's activity and operation membership;
4. deliver the replacement password privately.

WireGuard server private key or Terraform state:

1. stop student access;
2. generate a new server key pair;
3. update Terraform and regenerate every student profile;
4. distribute the new templates individually;
5. review GCP and application logs;
6. destroy and rebuild the lab if integrity is uncertain.

## Updates and teardown

Mythic is pinned through `mythic_version`. Test a new release in a disposable project, then
update it during a maintenance window. Do not track the upstream `main` branch.

For complete teardown:

```bash
cd terraform
terraform destroy
```

After teardown, verify that the reserved address and other billable resources were removed.
Retain or securely delete Terraform state, roster data, logs, and submissions according to the
institution's retention policy.
